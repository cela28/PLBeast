# Phase 05.1: Code Review Report

**Reviewed:** 2026-06-29T08:59:33Z
**Depth:** standard
**Files Reviewed:** 2
**Status:** issues_found

## Summary

Phase 05.1 rewrites PLBeast's rotation tracking engine from a 10Hz C_Timer poll to an event-driven CDM-based model matching AzortharionUI's architecture. The rewrite is structurally sound: the old poll ticker and snapshot-diff code is fully removed, the new `PollPackLeader` self-correcting state machine is correctly implemented, CDM cache rebuild events are properly wired, and SavedVariables migration handles legacy integer-to-string conversion correctly.

However, there is one critical bug where spec-change resets the internal prediction variable without updating the visible icon texture, and one bug in the rapid-beast-change branch that silently skips the icon update. Additionally, two state variables are written but never read (dead state), and the CDM cache tracks two spell IDs that are never queried by any consumer.

## Critical Issues

### CR-01: Spec change resets prediction variable but never updates icon texture

**File:** `PLBeast/PLBeast.lua:806-810`
**Issue:** When `ACTIVE_PLAYER_SPECIALIZATION_CHANGED` fires, the handler sets `nextBeastId = "boar"` and calls `SaveState()`, but does NOT call `SetNextBeastId("boar")`. `SetNextBeastId` is the only function that calls `root.tex:SetTexture(...)` to update the displayed icon. If a BM hunter switches to SV (both Pack Leader eligible), `RefreshVisibility` keeps the frame visible and wires the OnUpdate handler, but `PollPackLeader` only updates the icon on state transitions (ready-appear or ready-consumed). In the "off" phase with no beast ready, none of the transition branches fire, so the icon remains showing the pre-switch beast texture indefinitely until the next beast spawn.

The direct assignment `nextBeastId = "boar"` at line 808 also bypasses `SetNextBeastId`'s fallback guard (`or "boar"`) and texture update, making this a silent desync between `nextBeastId` (correct: "boar") and the displayed texture (stale: whatever beast was last shown).

**Fix:**
```lua
-- PLBeast.lua lines 806-810: replace direct assignment with SetNextBeastId call
elseif event == "ACTIVE_PLAYER_SPECIALIZATION_CHANGED" then
    ClearPackLeaderState()
    SetNextBeastId("boar")
    QueueVisibilityRefresh()
```

This replaces the three lines (`nextBeastId = "boar"` + `SaveState()`) with a single `SetNextBeastId("boar")` call, which sets `nextBeastId`, writes `DB.plNextBeastId`, AND updates `root.tex:SetTexture`. The `SaveState()` call becomes redundant since `SetNextBeastId` already writes to `DB.plNextBeastId` at line 182.

## Warnings

### WR-01: Rapid beast change during ready phase does not update icon

**File:** `PLBeast/PLBeast.lua:378-380`
**Issue:** In `PollPackLeader`, the `nowReady and hadReady` branch updates `plBeast = nowBeast` but does not call `SetNextBeastId(nowBeast.id)`. If a different beast becomes active during the ready phase (e.g., CDM data briefly overlaps during a rapid transition), the internal tracking updates but the icon texture is not refreshed. The comment says "defensive: update if beast changed rapidly" but the defense is incomplete -- it updates the state variable without updating the display.

The research notes that Azor's version handles this via a separate `plDirty` flag consumed by a display layer. PLBeast intentionally omits `plDirty` but does not compensate with a direct icon update in this branch.

While the game mechanic makes this scenario rare (typically only one beast-ready buff is active at a time), the fix is trivial and the current code is silently wrong when it does occur.

**Fix:**
```lua
elseif nowReady and hadReady then
    -- Still ready: update in case beast changed rapidly
    if nowBeast ~= plBeast then
        SetNextBeastId(nowBeast.id)
    end
    plBeast = nowBeast
```

### WR-02: `plHasCdBuff` and `plDirty` state variables are written but never read

**File:** `PLBeast/PLBeast.lua:115-116, 174-176, 376-377, 387`
**Issue:** `plHasCdBuff` is initialized at line 115, reset in `ClearPackLeaderState` at line 174, and set in `PollPackLeader` at line 376 -- but is never read by any function. `plDirty` is initialized at line 116, reset in `ClearPackLeaderState` at line 175, and set in `PollPackLeader` at lines 377 and 387 -- but is never read by any function. The comment at line 116 says "kept for future consumers" but dead state variables that are actively written to are misleading -- they suggest an incomplete implementation where something should be reading them.

This is a code quality issue that degrades maintainability: a future developer seeing `plHasCdBuff = false` in `ClearPackLeaderState` will assume it matters for correctness and be afraid to remove it.

**Fix:** Remove both variables entirely (declaration, all assignments, and comments), or add a clear `-- UNUSED:` prefix to the declarations if they are genuinely planned for a future phase. If future consumers need them, they can be re-added when the consumer exists.

### WR-03: CDM cache stale-frame false positive risk with no validation

**File:** `PLBeast/PLBeast.lua:290-298`
**Issue:** `CDMFrameHasAura` reads `data.cdmFrame.auraInstanceID` to detect whether a beast buff is active. If the CDM frame pool recycles `data.cdmFrame` to track a completely different spell, the frame's `auraInstanceID` could be a positive number from the new spell's aura -- causing a false positive where PLBeast thinks a beast is ready when it is not.

The lazy re-resolve at lines 292-293 only fires when `data.cdmFrame` is nil, not when the frame has been recycled to a different cooldown. There is no check that `GetFrameCooldownID(data.cdmFrame) == data.cooldownID` before trusting the `auraInstanceID`.

This matches Azor's implementation verbatim (per D-05), so it is a known architectural limitation rather than a PLBeast-specific regression. The `ScheduleRebuild` debounce (0.5s) triggered by CDM events should rebuild the cache shortly after any pool recycling occurs, limiting the false-positive window. However, during that 0.5s window, the icon could briefly show an incorrect beast.

**Fix:** Add a cooldownID ownership check before trusting the auraInstanceID:
```lua
local function CDMFrameHasAura(data)
    if not data then return false end
    if not data.cdmFrame and data.cooldownID then
        data.cdmFrame = FindCDMFrameByCooldownID(data.cooldownID)
    end
    if not data.cdmFrame then return false end
    -- Validate frame still owns the expected cooldown (pool recycling guard)
    if data.cooldownID and GetFrameCooldownID(data.cdmFrame) ~= data.cooldownID then
        data.cdmFrame = FindCDMFrameByCooldownID(data.cooldownID)
        if not data.cdmFrame then return false end
    end
    local instId = data.cdmFrame.auraInstanceID
    return type(instId) == "number" and instId > 0
end
```

## Info

### IN-01: TRACKED_SPELL_IDS includes spells never queried by any consumer

**File:** `PLBeast/PLBeast.lua:68-74`
**Issue:** `TRACKED_SPELL_IDS` includes `SPELL_HOTPL_PARENT` (471876) and `PL_COUNTDOWN` (471877). These cause `BuildCDMCache` to cache entries for these spell IDs, but `PollPackLeader` only iterates `PL_BEASTS` (boar/bear/wyvern spells) and never reads `cdmCache[SPELL_HOTPL_PARENT]` or `cdmCache[PL_COUNTDOWN]`. The cached entries consume memory and processing time during the build without any consumer.

The plan Step F explicitly mentioned reading these ("Read countdown buff from cdmCache[SPELL_HOTPL_PARENT] and cdmCache[PL_COUNTDOWN] via CDMFrameHasAura"), but the implementation simplified this away per the comment "no CD-buff display needed."

**Fix:** Remove `SPELL_HOTPL_PARENT` and `PL_COUNTDOWN` from `TRACKED_SPELL_IDS` since no code reads them. If a future phase needs CD-buff tracking, re-add them at that time:
```lua
local TRACKED_SPELL_IDS = {}
for _, beast in ipairs(PL_BEASTS) do
    TRACKED_SPELL_IDS[beast.spell] = true
end
```

### IN-02: `PL_COUNTDOWN` constant is declared but only used as a TRACKED_SPELL_IDS key

**File:** `PLBeast/PLBeast.lua:41, 70`
**Issue:** `PL_COUNTDOWN = 471877` is declared at line 41 and used only at line 70 to add it to `TRACKED_SPELL_IDS`. As noted in IN-01, the cached entry for this spell is never read. If IN-01 is applied, `PL_COUNTDOWN` becomes entirely unused dead code.

**Fix:** Remove `PL_COUNTDOWN` if IN-01 is applied. If kept for future use, add a comment: `-- Reserved for future CD-buff tracking`.
