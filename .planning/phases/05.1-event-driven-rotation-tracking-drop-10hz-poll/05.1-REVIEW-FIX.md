## Fixed Issues

### CR-01: Spec change resets prediction variable but never updates icon texture

**Files modified:** `PLBeast/PLBeast.lua`
**Commit:** a84aa81
**Applied fix:** Replaced direct `nextBeastId = "boar"` assignment + `SaveState()` with a single `SetNextBeastId("boar")` call. `SetNextBeastId` sets the variable, writes to `DB.plNextBeastId`, AND calls `root.tex:SetTexture(...)` to update the displayed icon -- eliminating the desync between internal state and displayed texture on spec change.

### WR-01: Rapid beast change during ready phase does not update icon

**Files modified:** `PLBeast/PLBeast.lua`
**Commit:** d28e76c
**Applied fix:** Added a `nowBeast ~= plBeast` guard in the `nowReady and hadReady` branch of `PollPackLeader` that calls `SetNextBeastId(nowBeast.id)` when the active beast changes. This ensures the icon texture is updated even during rapid beast transitions, completing the defensive update that previously only updated the state variable.

### WR-02: `plHasCdBuff` and `plDirty` state variables are written but never read

**Files modified:** `PLBeast/PLBeast.lua`
**Commit:** 4691d37
**Applied fix:** Removed both dead state variables (`plHasCdBuff`, `plDirty`) entirely -- declarations at lines 115-116, all assignments in `ClearPackLeaderState` and `PollPackLeader`. Left a comment at the former declaration site noting they were removed as dead state and can be re-added when a consumer exists.

### WR-03: CDM cache stale-frame false positive risk with no validation

**Files modified:** `PLBeast/PLBeast.lua`
**Commit:** 8dcb858
**Applied fix:** Added a cooldownID ownership check in `CDMFrameHasAura` before trusting `auraInstanceID`. If `GetFrameCooldownID(data.cdmFrame) ~= data.cooldownID`, the function re-resolves the frame via `FindCDMFrameByCooldownID`. This guards against CDM frame pool recycling where a frame assigned to a different spell could produce a false-positive aura detection during the 0.5s debounce window.
