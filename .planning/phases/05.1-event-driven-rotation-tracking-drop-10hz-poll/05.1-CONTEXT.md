**Gathered:** 2026-06-29

<domain>
## Phase Boundary

Replace PLBeast's constant 10Hz `C_Timer` poll with a per-frame CDM-polling model that behaviorally matches AzortharionUI's Pack Leader implementation. Adopt Azor's self-correcting prediction model (pin to the ready beast, advance on consumption), lazy CDM frame re-resolve, and debounced cache rebuild triggers. Drop the explicit login/boss-pull reset for Azor parity. The result: identical next-beast predictions step-for-step with AzortharionUI, verified by in-game side-by-side testing and AddonProfiler measurement.

**In scope (PERF-01):**
- Remove `C_Timer.NewTicker` poll (`StartPollTicker`/`StopPollTicker`, `POLL_INTERVAL`, `pollTicker`)
- Adopt Azor's per-frame OnUpdate detection via CDM frame state (not `UNIT_AURA`)
- Adopt Azor's self-correcting prediction model (`PollPackLeader` lines 245–326)
- Lazy per-`cooldownID` CDM frame re-resolve (`CDMFrameHasAura` → `FindCDMFrameByCooldownID`)
- Subscribe to Azor's 3 CDM rebuild triggers via debounced `ScheduleRebuild`
- `BuildCDMCache` on combat-enter if cache is empty (Azor's `PLAYER_REGEN_DISABLED` fallback)
- Drop login/boss-pull reset-to-wyvern; default to boar; persist across relog; reset to boar on spec change
- Remove the rebuild-on-all-false self-heal (superseded by lazy re-resolve)
- In-game side-by-side verification with AzortharionUI
- Performance measurement via AddonProfiler

**Out of scope:**
- Re-adding login/boss-pull reset (deferred to post-5.1 evaluation — see `reconsider-login-boss-reset` todo)
- Changes to the icon UI, options frame, or visibility gating
- Any behavior that diverges from AzortharionUI's Pack Leader implementation

<decisions>
## Implementation Decisions

### Detection Mechanism
- **D-01:** Match AzortharionUI's per-frame CDM poll exactly — `Hunter.Update()` pattern with `GetTime()` throttle (max once per game frame). Do NOT use `UNIT_AURA` as the detection trigger. This means an OnUpdate handler drives `PollPackLeader()`, reading CDM frame state via cached refs.
- **D-02:** Read aura state exclusively through CDM frame references (`data.cdmFrame.auraInstanceID`), matching Azor's approach. Remove the `C_UnitAuras.GetPlayerAuraBySpellID` aura-reading path from the detection loop. The CDM cache is the sole source of truth for beast ready buffs.

### CDM Cache & Rebuild Strategy
- **D-03:** Subscribe to all 3 CDM rebuild triggers, matching AzortharionUI exactly: `COOLDOWN_VIEWER_DATA_LOADED`, `COOLDOWN_VIEWER_SPELL_OVERRIDE_UPDATED`, and `EventRegistry: CooldownViewerSettings.OnDataChanged`. All route through a **0.5s debounced** `ScheduleRebuild` (matches Azor's `C_Timer.NewTimer(0.5, BuildCDMCache)`).
- **D-04:** Add `BuildCDMCache` on `PLAYER_REGEN_DISABLED` if cache is empty, matching Azor's combat-enter fallback (Azor `OnEvent` line 452–453).
- **D-05:** Replace the rebuild-whole-cache-on-all-false self-heal with Azor's **lazy per-frame re-resolve** — when a cached `cdmFrame` ref is nil but `cooldownID` is known, call `FindCDMFrameByCooldownID` on demand (Azor `CDMFrameHasAura` lines 203–213). This fixes the CDM pool rebuild freeze without polling.

### Prediction Model
- **D-06:** Adopt Azor's **self-correcting prediction** model: when a ready buff appears (`nowReady and not hadReady`), set `plNextBeastId = nowBeast.id` — pin to the ready beast itself. When the buff is consumed (`not nowReady and hadReady`), advance to `NEXT_BEAST[oldBeast.id]`. The icon answers "which beast is expected" and self-corrects whenever reality disagrees with the prediction.
- **D-07:** This changes visible behavior: current PLBeast shows the beast AFTER the ready one during the ready phase (e.g., boar ready → icon shows bear). Azor shows the ready beast itself (boar ready → icon shows boar). The user confirmed adopting Azor's semantic.

### Reset Semantics
- **D-08:** Default prediction is **boar** (Azor's `plNextBeastId = 'boar'` line 53). Spec change resets to boar (`DetectSpec(true)` → `plNextBeastId = 'boar'`, Azor line 436).
- **D-09:** **No login/boss-pull reset.** Prediction persists across relog via SavedVariables (Azor `SaveState`/`RestoreState`). The self-correcting model makes explicit resets largely redundant. Re-adding these resets is a separate post-5.1 decision (see deferred items).

### Rotation Order
- **D-10:** Rotation order already matches AzortharionUI: `boar → bear → wyvern → boar`. No change needed.

### Claude's Discretion
- How to restructure PLBeast's existing code to match Azor's `PollPackLeader` flow — the researcher should read Azor's full `Hunter.lua` and map it to PLBeast's architecture
- Whether to keep PLBeast's `ID_BY_INDEX`/`INDEX_BY_ID` tables or simplify to Azor's string-based `plNextBeastId` pattern
- How to handle the `TRACKED_SPELL_IDS` expansion — Azor tracks additional spells (PL_TALENT, PL_COUNTDOWN, PRECISE_SHOTS, etc.) that PLBeast doesn't need for beast prediction only
- Exact OnUpdate handler wiring — whether to use the existing `root` frame's OnUpdate or create a dedicated lightweight frame
