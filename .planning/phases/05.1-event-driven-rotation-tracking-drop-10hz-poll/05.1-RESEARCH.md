## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-01:** Match AzortharionUI's per-frame CDM poll exactly — `Hunter.Update()` pattern with `GetTime()` throttle (max once per game frame). Do NOT use `UNIT_AURA` as the detection trigger. This means an OnUpdate handler drives `PollPackLeader()`, reading CDM frame state via cached refs.
- **D-02:** Read aura state exclusively through CDM frame references (`data.cdmFrame.auraInstanceID`), matching Azor's approach. Remove the `C_UnitAuras.GetPlayerAuraBySpellID` aura-reading path from the detection loop. The CDM cache is the sole source of truth for beast ready buffs.
- **D-03:** Subscribe to all 3 CDM rebuild triggers, matching AzortharionUI exactly: `COOLDOWN_VIEWER_DATA_LOADED`, `COOLDOWN_VIEWER_SPELL_OVERRIDE_UPDATED`, and `EventRegistry: CooldownViewerSettings.OnDataChanged`. All route through a **0.5s debounced** `ScheduleRebuild` (matches Azor's `C_Timer.NewTimer(0.5, BuildCDMCache)`).
- **D-04:** Add `BuildCDMCache` on `PLAYER_REGEN_DISABLED` if cache is empty, matching Azor's combat-enter fallback (Azor `OnEvent` line 452-453).
- **D-05:** Replace the rebuild-whole-cache-on-all-false self-heal with Azor's **lazy per-frame re-resolve** — when a cached `cdmFrame` ref is nil but `cooldownID` is known, call `FindCDMFrameByCooldownID` on demand (Azor `CDMFrameHasAura` lines 203-213). This fixes the CDM pool rebuild freeze without polling.
- **D-06:** Adopt Azor's **self-correcting prediction** model: when a ready buff appears (`nowReady and not hadReady`), set `plNextBeastId = nowBeast.id` — pin to the ready beast itself. When the buff is consumed (`not nowReady and hadReady`), advance to `NEXT_BEAST[oldBeast.id]`. The icon answers "which beast is expected" and self-corrects whenever reality disagrees with the prediction.
- **D-07:** This changes visible behavior: current PLBeast shows the beast AFTER the ready one during the ready phase (e.g., boar ready → icon shows bear). Azor shows the ready beast itself (boar ready → icon shows boar). The user confirmed adopting Azor's semantic.
- **D-08:** Default prediction is **boar** (Azor's `plNextBeastId = 'boar'` line 53). Spec change resets to boar (`DetectSpec(true)` → `plNextBeastId = 'boar'`, Azor line 436).
- **D-09:** **No login/boss-pull reset.** Prediction persists across relog via SavedVariables (Azor `SaveState`/`RestoreState`). The self-correcting model makes explicit resets largely redundant. Re-adding these resets is a separate post-5.1 decision.
- **D-10:** Rotation order already matches AzortharionUI: `boar -> bear -> wyvern -> boar`. No change needed.

## Summary

Phase 05.1 replaces PLBeast's constant 10 Hz `C_Timer.NewTicker` poll with a per-frame OnUpdate CDM-polling model that exactly matches AzortharionUI's `Hunter.lua` Pack Leader implementation. The change touches five interlocked areas: (1) detection mechanism — OnUpdate throttled by `GetTime()` instead of a recurring ticker; (2) prediction model — self-correcting pin-to-ready-beast semantics replacing dead-reckoning; (3) CDM cache maintenance — three rebuild triggers plus lazy per-frame re-resolve instead of a rebuild-on-all-false fallback; (4) reset semantics — persist across relog, reset to boar on spec change, no login/boss-pull reset; (5) state persistence — `DB.plNextBeastId` string key instead of or alongside `DB.nextIndex`.

All decisions are locked in CONTEXT.md. The authoritative reference is `AzortharionUI/Modules/BuffTracking/Hunter.lua` (read in full during this research session from the repo zip). Every implementation decision maps to specific line numbers in that file. This is a refactor of `PLBeast/PLBeast.lua` only — no new files, no UI changes, no options frame changes.

The self-correcting model is a behavioral change the user has explicitly confirmed (D-07): during the ready window the icon will show the ready beast itself rather than the beast after it. This is the correct semantic and will match Azor step-for-step.

**Primary recommendation:** Implement `PollPackLeader()` as a near-direct Lua port of Azor's `Hunter.lua` lines 245-326, adapted to PLBeast's single-file architecture. Wire it from an OnUpdate on a dedicated tiny frame with `GetTime()` throttle. Keep PLBeast-specific simplifications: no `plPhase` tracking beyond 'off'/'ready', no `PL_TALENT`/`PL_COUNTDOWN` CDM tracking, `TRACKED_SPELL_IDS` stays as the three beast-ready spell IDs only.

## Architecture Patterns

### System Architecture Diagram

```
Game Frame (every ~16ms)
  └─ OnUpdate handler (dedicated pollFrame, created at module load)
       └─ OnPollUpdate()
            ├─ GetTime() throttle: skip if same game frame as last poll
            └─ PollPackLeader()
                 ├─ Guard: hasPackLeader? (set by BuildCDMCache on spec check)
                 ├─ Guard: cdmCacheBuilt?
                 ├─ Scan PL_BEASTS (boar → bear → wyvern, first active wins)
                 │    └─ CDMFrameHasAura(data)
                 │         ├─ Lazy re-resolve: if cdmFrame nil → FindCDMFrameByCooldownID(cooldownID)
                 │         └─ Return data.cdmFrame.auraInstanceID > 0
                 ├─ Self-correcting prediction:
                 │    ├─ nowReady and not hadReady → plNextBeastId = nowBeast.id
                 │    │    └─ SetNextBeastId(plNextBeastId), SaveState()
                 │    └─ not nowReady and hadReady → plNextBeastId = NEXT_BEAST[oldBeast.id]
                 │         └─ SetNextBeastId(plNextBeastId), SaveState()
                 └─ (no change if nowReady == hadReady)

CDM Rebuild Triggers (all 3 route through ScheduleRebuild → 0.5s debounced C_Timer)
  ├─ COOLDOWN_VIEWER_DATA_LOADED event
  ├─ COOLDOWN_VIEWER_SPELL_OVERRIDE_UPDATED event
  └─ EventRegistry: CooldownViewerSettings.OnDataChanged callback

Combat-Enter Fallback
  └─ PLAYER_REGEN_DISABLED → if not next(cdmCache) then BuildCDMCache() end

Spec Change
  └─ PLAYER_SPECIALIZATION_CHANGED / ACTIVE_PLAYER_SPECIALIZATION_CHANGED
       └─ plNextBeastId = 'boar', SaveState(), ScheduleRebuild()

SavedVariables Persistence
  ├─ SaveState(): DB.plNextBeastId = plNextBeastId  (on every prediction change)
  └─ RestoreState(): plNextBeastId = DB.plNextBeastId  (in ADDON_LOADED)
```

### Pattern 1: Per-Frame Throttle (Hunter.Update equivalent)

**What:** An OnUpdate handler that calls `PollPackLeader()` at most once per game frame, using `GetTime()` as the gate.

**When to use:** Whenever you need a polling loop that runs as fast as game frames allow but never redundantly within the same frame.

```lua
-- Source: Azor Hunter.lua lines 411-418 [VERIFIED: read from source]
local lastPolledTime = -1

local function OnPollUpdate()
    local now = GetTime()
    if now == lastPolledTime then return end
    lastPolledTime = now
    PollPackLeader()
end

-- Wire (at module load or PLAYER_LOGIN):
-- pollFrame:SetScript("OnUpdate", OnPollUpdate)
```

**PLBeast integration note:** The existing `root` frame has no OnUpdate. A dedicated `pollFrame = CreateFrame("Frame")` created at module load time is recommended — it avoids coupling to the root frame lifecycle (root is created at PLAYER_LOGIN) and is the cleanest approach.

### Pattern 2: Self-Correcting Prediction (PollPackLeader core)

**What:** Pin `plNextBeastId` to the ready beast when a buff appears; advance to `NEXT_BEAST[oldBeast.id]` when the buff is consumed.

**When to use:** This is the only prediction model in Phase 5.1.

```lua
-- Source: Azor Hunter.lua lines 245-326 (condensed to PLBeast scope) [VERIFIED: read from source]
local plPhase       = 'off'  -- 'off' | 'ready'
local plBeast       = nil    -- currently-ready beast data table ref

local function PollPackLeader()
    if not hasPackLeader then
        if plPhase ~= 'off' then
            plPhase = 'off'; plBeast = nil
        end
        return
    end
    if not cdmCacheBuilt then return end

    local hadReady = (plPhase == 'ready')
    local oldBeast = plBeast

    -- Scan PL_BEASTS in order: boar, bear, wyvern (first active wins)
    local nowReady = false
    local nowBeast = nil
    for _, beastData in ipairs(PL_BEASTS) do
        local data = cdmCache[beastData.spell]
        if data and CDMFrameHasAura(data) then
            nowReady = true
            nowBeast = beastData
            break
        end
    end

    if nowReady and not hadReady then
        plPhase       = 'ready'
        plBeast       = nowBeast
        plNextBeastId = nowBeast.id
        SaveState()
        SetNextBeastId(plNextBeastId)
    elseif nowReady and hadReady then
        plBeast = nowBeast  -- keep tracking (beast can shift mid-window)
    elseif not nowReady and hadReady then
        plPhase       = 'off'
        plBeast       = nil
        plNextBeastId = oldBeast and NEXT_BEAST[oldBeast.id] or 'boar'
        SaveState()
        SetNextBeastId(plNextBeastId)
    end
end
```

### Pattern 3: Lazy CDM Frame Re-Resolve (CDMFrameHasAura)

**What:** When a cache entry has a `cooldownID` but `cdmFrame` is nil, lazily re-resolve the frame on demand. This replaces the rebuild-on-all-false self-heal.

```lua
-- Source: Azor Hunter.lua lines 203-213 [VERIFIED: read from source]
local function CDMFrameHasAura(data)
    if not data then return false end
    if not data.cdmFrame and data.cooldownID then
        data.cdmFrame = FindCDMFrameByCooldownID(data.cooldownID)
    end
    if not data.cdmFrame then return false end
    local instId = data.cdmFrame.auraInstanceID
    return type(instId) == 'number' and instId > 0
end
```

**Key difference from current PLBeast:** Azor does NOT check `FrameStillOwnsCooldown` on every read. It trusts the frame ref until a rebuild trigger fires (which reassigns `data.cdmFrame` from the fresh cache). The lazy nil-resolve handles the stale-frame case sufficiently. PLBeast's existing `FrameStillOwnsCooldown` / `EnsureCacheFramesResolved` should be removed.

### Pattern 4: Debounced Cache Rebuild (ScheduleRebuild)

**What:** Cancel any pending rebuild timer and start a new 0.5s one-shot. All three CDM events route through this function.

```lua
-- Source: Azor Hunter.lua lines 171-177 [VERIFIED: read from source]
local rebuildTimer
local function ScheduleRebuild()
    if rebuildTimer and not rebuildTimer:IsCancelled() then
        rebuildTimer:Cancel()
    end
    rebuildTimer = C_Timer.NewTimer(0.5, BuildCDMCache)
end
```

### Pattern 5: State Persistence (SaveState / RestoreState)

**What:** Write `plNextBeastId` to `DB.plNextBeastId` on every prediction change; read it back on `ADDON_LOADED`.

```lua
-- Source: Azor Hunter.lua lines 221-234, adapted to PLBeast's DB pattern [VERIFIED: read from source]
local function SaveState()
    DB.plNextBeastId = plNextBeastId
end

local function RestoreState()
    local saved = DB.plNextBeastId
    if saved and NEXT_BEAST[saved] then
        plNextBeastId = saved
    end
end
```

**SavedVariables migration:** PLBeast currently persists `DB.nextIndex` (integer 1-3). Phase 5.1 adds `DB.plNextBeastId` (string). In `ADDON_LOADED`, after applying defaults, add a one-time migration:

```lua
-- Phase 5.1 migration: DB.nextIndex (integer) → DB.plNextBeastId (string) [ASSUMED - no existing installs in wild]
if DB.plNextBeastId == nil and DB.nextIndex then
    DB.plNextBeastId = ID_BY_INDEX[DB.nextIndex] or "boar"
end
```

### Anti-Patterns to Avoid

- **Keeping the `UNIT_AURA` detection path:** D-02 locks CDM frames as the sole source of truth. `IsReadyViaAura`, `IsReadyBuffActive`, and `RegisterEvent("UNIT_AURA")` must all be removed.
- **Checking `FrameStillOwnsCooldown` on every poll tick:** Azor's model trusts the cached frame ref until a rebuild trigger fires. Remove `FrameStillOwnsCooldown` and `EnsureCacheFramesResolved`.
- **Rebuild-on-all-false self-heal:** The entire block at PLBeast.lua lines 398-410 is superseded by `CDMFrameHasAura`'s lazy nil-resolve. Remove it.
- **Calling `SyncNextFromAddedReady` in the new model:** The old advancement function advances to `NEXT_BEAST[beastId]` (the beast after the ready one). The new model pins to the ready beast itself. `SyncNextFromAddedReady` must not be called from `PollPackLeader`.
- **Wiring OnUpdate before PLAYER_LOGIN if using the root frame:** Root frame doesn't exist until PLAYER_LOGIN. Use a dedicated pollFrame created at module load time to avoid ordering constraints.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Debounced timer | Manual `GetTime()` accumulator + state flag | `C_Timer.NewTimer(0.5, fn)` with `Cancel()` / `IsCancelled()` | WoW's timer API handles this; already used by PLBeast for other deferred calls |
| CDM category enumeration | Hardcoded integer array `{ 0, 1, 2, 3, 4, 5 }` | `Enum.CooldownViewerCategory` enum values (Azor uses these) | Enum names are more readable and correct; integer fallback is acceptable if enum unavailable |
| Frame tree DFS | Custom queue or iteration | `frame:GetChildren()` in recursive scan (already in `FindCDMFrameByCooldownID`) | Already implemented and working; don't rebuild |

**Key insight:** The core implementation challenge of Phase 5.1 is not writing new algorithms — it is correctly removing old code and wiring the existing pieces (`FindCDMFrameByCooldownID`, `BuildCDMCache`, `SetNextBeastId`) in Azor's pattern. The behavioral change (prediction model) is about 30 lines of new code in `PollPackLeader`.

## Common Pitfalls

### Pitfall 1: OnUpdate fires before root frame exists

**What goes wrong:** If `PollPackLeader` is wired to the root frame's OnUpdate but the root frame is created at `PLAYER_LOGIN`, any pre-login OnUpdate fire will panic on nil root.

**Why it happens:** Module-level code runs at file load (before PLAYER_LOGIN). A dedicated polling frame created at module load time and calling `PollPackLeader` will run before `CreateBeastIcon()`.

**How to avoid:** Use a dedicated `pollFrame = CreateFrame("Frame")` at module load time. Guard `PollPackLeader` against nil root (PLBeast already guards `root` before accessing `root.tex`). The guard is: `if not root then return end` before calling `SetNextBeastId`.

**Warning signs:** Nil-access error on `root.tex` or `root:SetShown()` during login.

### Pitfall 2: Default remains "wyvern" — must change to "boar"

**What goes wrong:** `defaults.nextIndex = 1` currently maps to "wyvern". D-08 locks the default to "boar". If not changed, first-time users diverge from Azor.

**Why it happens:** PLBeast.lua line 84 comment: "Note: Default fallback is 'wyvern' (PLH uses 'boar'; intentional PLBeast deviation)." Phase 5.1 reverses this.

**How to avoid:** Add `defaults.plNextBeastId = "boar"` and change `defaults.nextIndex = 2`. Update `nextBeastId = "boar"` at module-level initialization.

**Warning signs:** First-load icon shows wyvern; Azor shows boar.

### Pitfall 3: UNIT_AURA still registered after removing detection path

**What goes wrong:** `RefreshVisibility()` currently registers/unregisters `UNIT_AURA` when Pack Leader activates/deactivates. If this remains but the handler does nothing useful, it wastes event dispatch per every player aura change.

**How to avoid:** Remove all `RegisterEvent("UNIT_AURA")` and `UnregisterEvent("UNIT_AURA")` calls from `RefreshVisibility()`. Remove the `UNIT_AURA` branch in `OnEvent`.

**Warning signs:** Debug log shows CheckAuraState or PollPackLeader firing on irrelevant aura events.

### Pitfall 4: Old prediction semantic left alive via `SyncNextFromAddedReady`

**What goes wrong:** `SyncNextFromAddedReady` advances to `NEXT_BEAST[beastId]` (beast AFTER the ready one). If called anywhere in the new flow, prediction is off by one.

**How to avoid:** `SyncNextFromAddedReady` must be removed or left as dead code (not called). The new `PollPackLeader` directly assigns `plNextBeastId = nowBeast.id` — no call to `SyncNextFromAddedReady`.

**Warning signs:** Icon shows bear when boar is ready (old dead-reckoning semantic: boar ready → advance to bear).

### Pitfall 5: DB.nextIndex not migrated for existing users

**What goes wrong:** Existing character saves have `DB.nextIndex = 2` (boar) but `DB.plNextBeastId` will be nil. `RestoreState()` finds nil and leaves prediction at the module default.

**How to avoid:** In `ADDON_LOADED`, after applying defaults, check and migrate:
```lua
if DB.plNextBeastId == nil and DB.nextIndex then
    DB.plNextBeastId = ID_BY_INDEX[DB.nextIndex] or "boar"
end
```

**Warning signs:** User reports icon always shows boar after update regardless of saved state.

### Pitfall 6: rebuildTimer nil guard missing

**What goes wrong:** On first call to `ScheduleRebuild`, `rebuildTimer` is nil. `rebuildTimer:Cancel()` panics.

**How to avoid:** Use Azor's guard exactly: `if rebuildTimer and not rebuildTimer:IsCancelled() then rebuildTimer:Cancel() end`. [VERIFIED: read from Azor Hunter.lua line 173]

### Pitfall 7: Spec-change event name — register both for safety

**What goes wrong:** Azor registers `PLAYER_SPECIALIZATION_CHANGED`. PLBeast registers `ACTIVE_PLAYER_SPECIALIZATION_CHANGED`. In WoW 12.x retail both exist but may fire on different conditions; using only one might miss a spec-change notification.

**How to avoid:** Register both events and route both to the same spec-change handler (reset to boar, ScheduleRebuild). This is the safe approach pending in-game verification. [ASSUMED — cannot verify which event fires reliably on 12.x without in-game testing]

**Warning signs:** Spec change in-game does not reset icon to boar.

### Pitfall 8: `PLAYER_ENTERING_WORLD` and `ENCOUNTER_START` handlers still registered

**What goes wrong:** These events are currently registered in `eventFrame` at module-level (`RegisterEvent("PLAYER_ENTERING_WORLD")` line 824, `RegisterEvent("ENCOUNTER_START")` line 825). After Phase 5.1 removes the reset handlers for those events, the events are still fired to `OnEvent`. The branches must be removed from `OnEvent`; the `RegisterEvent` calls must also be removed to avoid unnecessary dispatch.

**How to avoid:** Remove both `RegisterEvent` calls and the corresponding `OnEvent` branches. [VERIFIED: read from PLBeast.lua source, lines 824-825 and 957-974]

## Open Questions

1. **Should PLBeast track `PL_TALENT` (471876) and `PL_COUNTDOWN` (471877) in its CDM cache?**
   - What we know: Azor tracks these for "ticking" phase display. `hasPackLeader` in Azor is set via `IsPlayerSpell(PL_TALENT)` inside `BuildCDMCache` — a direct API call, NOT a CDM cache lookup.
   - Recommendation: PLBeast does NOT need `PL_TALENT` / `PL_COUNTDOWN` in `TRACKED_SPELL_IDS`. Set `hasPackLeader` via `IsPlayerSpell(SPELL_HOTPL_PARENT)` inside `BuildCDMCache` (same as Azor). Keep `TRACKED_SPELL_IDS` as the three beast-ready spell IDs only.

2. **Dedicated polling frame vs. extending root frame's OnUpdate?**
   - Recommendation: Dedicated `pollFrame = CreateFrame("Frame")` created at module load time. Zero visual cost, clean separation, no ordering dependency on root frame creation. This is Claude's discretion.

3. **Should `ID_BY_INDEX` / `INDEX_BY_ID` be removed?**
   - Recommendation: Keep them for the `DB.nextIndex` migration and for the `/plbeast reset` slash command which still calls `SetNextBeastId`. `INDEX_BY_ID` is used inside `SetNextBeastId` to keep `DB.nextIndex` in sync (for backward compat). These tables are harmless and removing them adds risk. Claude's discretion.
