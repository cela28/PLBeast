## Summary

Phase 5.1 is a surgical refactor of PLBeast's detection engine. The goal is to replace the current `C_Timer.NewTicker(0.1, CheckAuraState)` with AzortharionUI's CDM-poll model, adopted verbatim from `Hunter.lua`. No UI, options, or visibility logic changes.

Reading the actual AzortharionUI `Hunter.lua` code (extracted from `AzortharionUI - 7.5.zip`) reveals a key architectural fact: **Azor does NOT use `UNIT_AURA` for Pack Leader detection at all.** The detection is a CDM state read driven by Azor's Scheduler — a hidden frame woken via `C_Timer.After`, calling `tracker.Update()` at `interval=1` (1 second). Inside `Hunter.Update()`, a `GetTime()` guard (`if now == lastPolledTime then return end`) prevents double execution within the same second. The net effect: `PollPackLeader()` runs at most once per second with zero cost between calls.

For PLBeast (which lacks Azor's Scheduler infrastructure), the equivalent is: attach `root:SetScript("OnUpdate", ...)` with the same `GetTime()` guard. Since `root` is shown/hidden by `RefreshVisibility()` based on `isPackLeaderActive`, the OnUpdate automatically ceases when Pack Leader is inactive — no explicit start/stop logic needed. The old `StartPollTicker`/`StopPollTicker` are removed entirely.

The self-correcting prediction model replaces `CheckAuraState`'s advance-on-added diff with Azor's pin-on-ready / advance-on-consumption state machine. The idle-safe CDM re-resolve replaces the all-false rebuild loop with `CDMFrameHasAura`'s lazy per-call `FindCDMFrameByCooldownID` fallback.

**Primary recommendation:** Adopt `Hunter.lua`'s `PollPackLeader()` logic verbatim, adapted for PLBeast's single-file structure. Wire it to `root`'s `OnUpdate` with a `GetTime()` guard. Add the three CDM rebuild event registrations. Total scope: ~150 lines changed in `PLBeast.lua` + 1 locale string update. The `PL_BEASTS` iterable array structure (Azor pattern) is recommended to replace PLBeast's four separate lookup tables.

### Pattern 1: Self-Correcting Prediction State Machine (PollPackLeader)

**What:** `PollPackLeader()` pins `plNextBeastId` to the actively-ready beast, then advances to `NEXT_BEAST[beastId]` when that buff is consumed. If reality disagrees with the prediction (e.g., after a /reload mid-rotation), the next ready buff fires and self-corrects immediately.

**When to use:** Called on every OnUpdate tick (once per second via GetTime guard). Replaces `CheckAuraState`.

```lua
-- Source: AzortharionUI/Modules/BuffTracking/Hunter.lua lines 245-326
-- PLBeast simplification: omit CD-buff tracking (PL_TALENT/PL_COUNTDOWN) since PLBeast
-- only displays next beast. plPhase is still needed for transition detection.
-- plDirty flag omitted: PLBeast updates icon directly in SetNextBeastId (no display layer).
local plPhase = 'off'   -- 'ready' | 'off'
local plBeast = nil     -- beastData entry when in 'ready' phase; used for NEXT_BEAST lookup

local function PollPackLeader()
    if not cdmCacheBuilt then return end

    local hadReady = (plPhase == 'ready')
    local oldBeast = plBeast

    local nowReady = false
    local nowBeast = nil
    for _, beastData in ipairs(PL_BEASTS) do
        local data = cdmCache[beastData.spell]
        if data and CDMFrameHasAura(data) then
            nowReady = true
            nowBeast = beastData
            break
        end
    end

    if nowReady and not hadReady then
        plPhase = 'ready'
        plBeast = nowBeast
        SetNextBeastId(nowBeast.id)   -- pin to ready beast (D-06, D-07)
    elseif nowReady and hadReady then
        plBeast = nowBeast            -- defensive: update if beast changed rapidly
    elseif not nowReady and hadReady then
        plPhase = 'off'
        plBeast = nil
        local nextId = (oldBeast and NEXT_BEAST[oldBeast.id]) or 'boar'
        SetNextBeastId(nextId)        -- advance to next beast (D-06)
    end
end
```

### Pattern 2: OnUpdate with GetTime() Throttle

**What:** Attach `OnUpdate` to `root` frame; guard with `GetTime()` to execute `PollPackLeader` at most once per second. Since `root` visibility is controlled by `isPackLeaderActive`, no explicit start/stop needed.

**Key discovery from reading Azor's code:** `GetTime()` in WoW returns 1-second resolution. The guard `if now == lastPolledTime then return end` fires at most once per second, matching Azor's Scheduler `interval=1` behavior exactly. This is NOT "once per game frame" — it is "once per second".

```lua
-- Source: AzortharionUI/Modules/BuffTracking/Hunter.lua lines 411-418 (adapted)
-- Wire in CreateBeastIcon() immediately after "root = f":
local lastPolledTime = -1
root:SetScript("OnUpdate", function()
    local now = GetTime()
    if now == lastPolledTime then return end
    lastPolledTime = now
    PollPackLeader()
end)
-- No StartPollTicker/StopPollTicker needed.
-- root:SetShown(false) in RefreshVisibility() stops OnUpdate automatically.
```

### Pattern 3: CDMFrameHasAura with Lazy Re-Resolve (D-05)

**What:** Check `cdmFrame.auraInstanceID > 0` for ready detection. If `cdmFrame` is nil (category-scan-only entry) or stale (CDM pool recycled), call `FindCDMFrameByCooldownID` on-demand. This replaces PLBeast's `EnsureCacheFramesResolved()` loop + `FrameStillOwnsCooldown()` pattern.

```lua
-- Source: AzortharionUI/Modules/BuffTracking/Hunter.lua lines 203-214
local function CDMFrameHasAura(data)
    if not data then return false end
    if not data.cdmFrame and data.cooldownID then
        data.cdmFrame = FindCDMFrameByCooldownID(data.cooldownID)
    end
    if not data.cdmFrame then return false end
    local instId = data.cdmFrame.auraInstanceID
    return type(instId) == 'number' and instId > 0
end
```

### Pattern 4: ScheduleRebuild Debounce (D-03)

**What:** Cancel-and-restart a 0.5s one-shot timer on each CDM event. Prevents `BuildCDMCache` from firing multiple times during a burst of CDM events (e.g., settings panel toggle).

```lua
-- Source: AzortharionUI/Modules/BuffTracking/Hunter.lua lines 171-177
local rebuildTimer
local function ScheduleRebuild()
    if rebuildTimer and not rebuildTimer:IsCancelled() then
        rebuildTimer:Cancel()
    end
    rebuildTimer = C_Timer.NewTimer(0.5, BuildCDMCache)
end
```

### Pattern 5: MatchTrackedSpell for Robust CDM Lookup

**What:** When resolving a cooldownID to a spellID during `BuildCDMCache`, check `info.spellID`, `info.overrideSpellID`, AND `info.linkedSpellIDs`. PLBeast currently only checks `info.spellID` via `ReadSpellFromCooldownID`. Adopting the full check closes a potential miss for override-registered spells.

```lua
-- Source: AzortharionUI/Modules/BuffTracking/Hunter.lua lines 77-89
local function MatchTrackedSpell(info)
    if not info then return nil end
    if info.spellID and TRACKED_SPELL_IDS[info.spellID] then return info.spellID end
    if info.overrideSpellID and TRACKED_SPELL_IDS[info.overrideSpellID] then
        return info.overrideSpellID
    end
    if info.linkedSpellIDs then
        for _, id in ipairs(info.linkedSpellIDs) do
            if TRACKED_SPELL_IDS[id] then return id end
        end
    end
    return nil
end
```

### Anti-Patterns to Avoid

- **Rebuild-on-all-false loop:** PLBeast's current `if not (current.wyvern or current.boar or current.bear) then BuildCDMCache()` fires on every tick during rotation downtime. Azor's `CDMFrameHasAura` lazy re-resolve handles stale frames per-read without a full rebuild scan. Remove this loop entirely.
- **Using UNIT_AURA as detection trigger:** Beast-ready buffs are not reliably delivered via UNIT_AURA. Azor proved CDM-only is sufficient and cheaper. Do not re-add UNIT_AURA to the detection path (D-02).
- **C_Timer.NewTicker for polling:** Tickers fire on a fixed interval regardless of frame visibility. An `OnUpdate` on a hidden frame fires zero times; a ticker fires unconditionally. Use OnUpdate on the icon frame instead.
- **plDirty flag:** Azor uses `plDirty` to coalesce changes before the display layer re-renders. PLBeast has no separate display layer — `SetNextBeastId` updates the texture synchronously. Do not add `plDirty`.
- **Keeping both `DB.nextIndex` and `DB.plNextBeastId`:** Dual-writing both creates maintenance confusion. Keep `DB.nextIndex` in the SavedVariables file (don't nil it — avoid churn), but stop writing to it after migration. `DB.plNextBeastId` is the sole source going forward.

## Common Pitfalls

### Pitfall 1: GetTime() Resolution — "Once Per Frame" is Actually "Once Per Second"

**What goes wrong:** `GetTime()` in WoW returns whole-second values. For roughly 60 consecutive `OnUpdate` calls within a second, the guard `if now == lastPolledTime then return end` blocks all but the first. The effective poll rate is ~1Hz, not per-frame.

**Why it happens:** WoW's `GetTime()` has 1-second resolution. This is intentional — Azor's Scheduler also uses `interval=1` for the PackLeader tracker, matching this behavior exactly.

**How to avoid:** This is correct and desired. Do NOT change the guard to `GetTime() - lastPolledTime < 0.05` — that would increase polling frequency to 20Hz and defeat the purpose.

**Warning signs:** If `PollPackLeader` is observed firing more than once per second, `lastPolledTime` is being reset incorrectly.

### Pitfall 2: root OnUpdate Only Fires When Frame Is Shown

**What goes wrong:** `root:SetScript("OnUpdate", fn)` does NOT fire when `root:IsShown()` returns false. This is the desired behavior (zero cost when Pack Leader inactive), but if a developer wires the poll to a separate always-visible frame, it runs even when `isPackLeaderActive` is false.

**Why it happens:** WoW only calls `OnUpdate` on visible frames.

**How to avoid:** Always wire `PollPackLeader` to the `root` frame's `OnUpdate`. Add a defensive guard at the top of `PollPackLeader` in case `cdmCacheBuilt` is false.

**Warning signs:** PollPackLeader being called when isPackLeaderActive is false.

### Pitfall 3: ENCOUNTER_START / Login Reset Branch Left as Dead Code

**What goes wrong:** If `ENCOUNTER_START` is unregistered but left in the `elseif` chain (or vice versa), no error occurs — the handler silently skips. But if the event IS still registered and the handler still calls `ResetRotationToWyvern`, the reset happens in violation of D-09.

**How to avoid:** Remove BOTH the event registration AND the handler branch for `ENCOUNTER_START`. Remove ONLY the reset-to-wyvern code from the `PLAYER_ENTERING_WORLD` handler (keep the event registered; the branch becomes a no-op or is removed).

**Warning signs:** After the phase, `ResetRotationToWyvern` should not be callable from any event path.

### Pitfall 4: SavedVariables Migration Order — Defaults Applied Before Migration

**What goes wrong:** The flat defaults merge applies `plNextBeastId = "boar"` when `PLBeastDB.plNextBeastId == nil`. If the migration runs AFTER this merge, `DB.plNextBeastId` is already set to `"boar"` and the migration's `if DB.plNextBeastId == nil` check never fires. All existing users start from boar regardless of their saved `nextIndex`.

**How to avoid:** Run the migration check on `PLBeastDB` (pre-merge) BEFORE the flat defaults merge, or check `PLBeastDB.nextIndex ~= nil` rather than `PLBeastDB.plNextBeastId == nil` (since defaults apply to the merged DB, not the pre-merge raw table).

```lua
-- CORRECT order in ADDON_LOADED:
-- 1. Migration (check raw PLBeastDB BEFORE merge)
if PLBeastDB.nextIndex ~= nil and PLBeastDB.plNextBeastId == nil then
    local map = { [1] = "wyvern", [2] = "boar", [3] = "bear" }
    PLBeastDB.plNextBeastId = map[PLBeastDB.nextIndex] or "boar"
end
-- 2. Flat defaults merge
for k, v in pairs(defaults) do
    if PLBeastDB[k] == nil then PLBeastDB[k] = v end
end
-- 3. DB = PLBeastDB
-- 4. RestoreState() reads DB.plNextBeastId -> plNextBeastId
```

### Pitfall 5: Removing prevReady and SeedAuraSnapshot as Separate Steps

**What goes wrong:** `SeedAuraSnapshot()` writes to `prevReady`. If `prevReady` is removed but `SeedAuraSnapshot()` is still called somewhere (e.g., in `RefreshVisibility`), a nil-index error occurs at runtime.

**How to avoid:** Remove all three as a unit: `prevReady` table declaration, `SeedAuraSnapshot()` definition, and all call sites of `SeedAuraSnapshot()` (currently in `RefreshVisibility` and `/plbeast reset` handler).

**Warning signs:** Any `prevReady` or `SeedAuraSnapshot` reference remaining after the phase.

### Pitfall 6: CDM Enum Guard

**What goes wrong:** If `Enum.CooldownViewerCategory` is nil at load time, `CDM_CATS` remains nil and `BuildCDMCache`'s category sweep is silently skipped. PLBeast's current hardcoded integer indexes `{0,1,2,3,4,5}` avoid this at the cost of less precision.

**How to avoid:** Adopt Azor's guard: `local cats = Enum and Enum.CooldownViewerCategory`. If nil, skip the category sweep (fall through to frame-tree scan only).

## State of the Art

| Old Approach | Current Approach (post-5.1) | Impact |
|--------------|----------------------------|--------|
| `C_Timer.NewTicker(0.1, CheckAuraState)` (10Hz) | `root:SetScript("OnUpdate")` + `GetTime()` guard (~1Hz) | ~10x fewer PollPackLeader calls; zero cost when hidden |
| `prevReady` diff table + advance-on-added | `plPhase == 'ready'` / not-ready state machine | Self-corrects without a saved snapshot; simpler logic |
| `EnsureCacheFramesResolved()` bulk loop + `FrameStillOwnsCooldown()` | `CDMFrameHasAura()` lazy per-call re-resolve | Re-resolve runs only for the one stale frame, not all entries |
| `C_UnitAuras.GetPlayerAuraBySpellID` fallback | CDM-only path | Removes secondary API call; CDM is the actual data source |
| Rebuild-on-all-false self-heal loop | `CDMFrameHasAura` lazy re-resolve + 3 event triggers | No rate-limited rebuild loop at idle; CDM events drive rebuilds |
| `DB.nextIndex` (integer 1–3) | `DB.plNextBeastId` (string "boar"/"bear"/"wyvern") | Matches Azor; self-documenting; no index-to-string translation |
| Default beast: wyvern | Default beast: boar (D-08) | Azor parity; self-correcting model makes default less critical |

**Deprecated patterns (to remove in this phase):**

- `NormalizeNextIndex()` — integer clamping for `DB.nextIndex`; not needed with string IDs
- `ID_BY_INDEX`, `INDEX_BY_ID` — index/string mapping tables; not needed with string-native model
- `SyncNextFromAddedReady()` — advance-on-added with start-time sort; replaced by PollPackLeader
- `ResetRotationToWyvern()`, `ResetAuraState()` — explicit resets; D-08/D-09 drop them
- `SeedAuraSnapshot()` — seeds prevReady at login; no prevReady in new model
- `IsReadyViaAura()`, `IsReadyBuffActive()` — C_UnitAuras fallback; D-02 removes it
- `FrameStillOwnsCooldown()`, `EnsureCacheFramesResolved()` — stale-frame detection; replaced by CDMFrameHasAura

## Open Questions

1. **Should `plDirty` flag be adopted from Azor?**
   - What we know: Azor uses `plDirty` to coalesce multiple state changes before the display layer re-renders. PLBeast updates the icon texture synchronously inside `SetNextBeastId`.
   - Recommendation: Skip `plDirty` for PLBeast. The display is a single `SetTexture` call — no coalescing needed.

2. **Should the `PLAYER_ENTERING_WORLD` event remain registered?**
   - What we know: The login-reset branch is removed per D-09. No other handlers exist for this event currently.
   - Recommendation: Keep the event registered with an empty handler body (or remove the handler branch entirely). Preserves registration for potential future use without risk.

3. **Should `BEAST_LABEL_BY_ID` be kept or derived from `PL_BEASTS`?**
   - What we know: Used only in `dprint` calls for debug output.
   - Recommendation: Keep as a separate table for readability, OR derive from `PL_BEASTS` (add a `label` field). Either works; consistency with Azor's structure favors the `PL_BEASTS` approach.
