# Phase 05.1: Event-Driven Rotation Tracking — Research

**Researched:** 2026-06-29
**Status:** Complete
**Requirement:** PERF-01

## Executive Summary

Phase 05.1 replaces PLBeast's 10Hz `C_Timer.NewTicker` poll with AzortharionUI's per-frame OnUpdate detection model. The transformation is well-defined: Azor's `Hunter.lua` (475 lines) serves as the authoritative reference. PLBeast's `PLBeast.lua` (975 lines) needs ~7 areas changed. The core insight: Azor uses per-frame `OnUpdate` with a `GetTime()` throttle (max once per game frame, ~16ms), NOT pure event-driven UNIT_AURA. The improvement over PLBeast's current approach is eliminating the *timer-based* 10Hz poll — detection still runs per-frame but only when the game renders, with zero overhead when idle/out-of-combat.

## AzortharionUI Reference Analysis

### Detection Model: `Hunter.Update()` (lines 411–418)

```lua
local lastPolledTime = -1
function Hunter.Update()
    local now = GetTime()
    if now == lastPolledTime then return end
    lastPolledTime = now
    PollTipStacks()
    PollPackLeader()
end
```

Called from an `OnUpdate` handler on AUI's main frame. The `GetTime()` equality check ensures `PollPackLeader()` runs at most once per game frame (~60 Hz at 60 FPS). No timer ticks when the frame isn't updating (e.g., loading screens, minimized client).

**Key difference from PLBeast:** PLBeast uses `C_Timer.NewTicker(0.1, ...)` which fires at a fixed 10Hz regardless of frame rendering. Azor's OnUpdate is frame-rate-adaptive and has zero cost when the frame isn't shown.

### Self-Correcting Prediction: `PollPackLeader()` (lines 245–326)

State variables (lines 51–58):
- `plPhase`: `'off'` | `'ticking'` | `'ready'` — three-state machine
- `plBeast`: the beast data table when a ready buff is active, nil otherwise
- `plNextBeastId`: `'boar'` | `'bear'` | `'wyvern'` — the predicted next beast (string-based)
- `plHasCdBuff`: whether the PL countdown/parent buff is active
- `plDirty`: change flag for UI consumers

Core logic:
1. **CDM read phase** (lines 253–291): Read Pack Leader countdown buff (`PL_TALENT`/`PL_COUNTDOWN`) and beast ready buffs (`PL_BEASTS[].spell`) from CDM frames via `CDMFrameHasAura()`.
2. **Transition logic** (lines 293–325):
   - Ready buff appears (`nowReady and not hadReady`): Set `plNextBeastId = nowBeast.id` — pin to the ready beast. Save state.
   - Ready buff persists (`nowReady and hadReady`): Update `plBeast` reference only (no prediction change).
   - Ready buff consumed (`not nowReady and hadReady`): Advance `plNextBeastId = NEXT_BEAST[oldBeast.id]`. Save state.
   - Countdown buff transitions: Update `plPhase` accordingly.

**Key semantic change:** PLBeast currently shows the beast AFTER the ready one (boar ready → icon shows bear via `NEXT_BEAST[boar]`). Azor shows the ready beast itself (boar ready → icon shows boar). The advance happens on consumption, not on appearance.

### CDM Cache: `BuildCDMCache()` (lines 137–169)

Two-phase scan matching PLBeast's existing pattern:
1. **Frame tree scan**: Recursive `ScanDeep()` up to depth 3 over `CDM_FRAME_NAMES` (`BuffBarCooldownViewer`, `BuffIconCooldownViewer`, `CDMGroups_Buffs`).
2. **Category set scan**: Iterates `CDM_CATS` (Essential, Utility, TrackedBuff, TrackedBar) via `GetCategorySet()`. Category entries get `cdmFrame = nil` (resolved lazily).

Cache key: `spellID → { cdmFrame, cooldownID }`.

**Difference:** Azor uses `ScanDeep()` (recursive DFS, depth cap 3) vs PLBeast's `GatherCooldownsFromFrameTree()` (iterative BFS, depth cap 4). Both achieve the same result. PLBeast also uses `ReadSpellFromCooldownID()` with pcall guard — Azor uses `MatchTrackedSpell()` with `spellID`/`overrideSpellID`/`linkedSpellIDs` matching.

### Lazy Re-Resolve: `CDMFrameHasAura()` (lines 203–214)

```lua
local function CDMFrameHasAura(data)
    if not data then return false, nil end
    if not data.cdmFrame and data.cooldownID then
        data.cdmFrame = FindCDMFrameByCooldownID(data.cooldownID)
    end
    if not data.cdmFrame then return false, nil end
    local instId = data.cdmFrame.auraInstanceID
    if instId and type(instId) == 'number' and instId > 0 then
        return true, instId
    end
    return false, nil
end
```

This replaces PLBeast's `EnsureCacheFramesResolved()` (which pre-resolves ALL entries) and the `rebuild-on-all-false` self-heal (which rebuilds the entire cache when no beast shows ready). Azor re-resolves per-entry, on demand, only when the cached frame ref is nil. Combined with `ScheduleRebuild()`, this handles CDM pool recycling without a full rebuild per check.

**Important:** Azor does NOT check `FrameStillOwnsCooldown()` — if a cached `cdmFrame` has a stale reference (pool recycled), `auraInstanceID` reads go nil and `CDMFrameHasAura` returns false. The lazy re-resolve triggers only when `cdmFrame` is nil, not when it's stale. The debounced `ScheduleRebuild` from CDM data-change events eventually rebuilds the cache and fixes stale refs. PLBeast's `FrameStillOwnsCooldown()` check is more defensive but also more expensive per-frame.

### Debounced Rebuild: `ScheduleRebuild()` (lines 172–177)

```lua
local rebuildTimer
local function ScheduleRebuild()
    if rebuildTimer and not rebuildTimer:IsCancelled() then
        rebuildTimer:Cancel()
    end
    rebuildTimer = C_Timer.NewTimer(0.5, BuildCDMCache)
end
```

Cancels any pending timer and schedules a new one at 0.5s. PLBeast's current `CooldownViewerSettings.OnDataChanged` handler already does this with `C_Timer.After(0.5, ...)` but only for that one event.

### Persistence: `SaveState()`/`RestoreState()` (lines 221–234)

Saves/loads `plNextBeastId` (string) to `AUI.db.char.plNextBeastId`. PLBeast currently persists `DB.nextIndex` (integer 1–3) and derives `nextBeastId` from `ID_BY_INDEX`. The migration path: switch to `DB.plNextBeastId` (string), add backward compat for old integer format.

### Reset Semantics: `DetectSpec()` (lines 424–442)

On spec change (`isSpecChange=true`): `plNextBeastId = 'boar'`, `SaveState()`. On login (`Initialize()`, line 461): `RestoreState()` then `DetectSpec(false)` — does NOT reset to boar, just clears state and rebuilds cache. Default initial value: `plNextBeastId = 'boar'` (line 53).

**No login reset. No boss-pull reset.** The self-correcting model auto-fixes stale predictions when a beast ready buff actually appears.

### Event Registrations (lines 460–475)

| Event | Handler | Purpose |
|-------|---------|---------|
| `PLAYER_SPECIALIZATION_CHANGED` | `DetectSpec(true)` | Spec change → reset to boar |
| `PLAYER_TALENT_UPDATE` | `UpdateTalents()` + `ScheduleRebuild()` | Talent changes |
| `TRAIT_CONFIG_UPDATED` | same | Talent config changes |
| `PLAYER_REGEN_DISABLED` | `BuildCDMCache()` if empty | Combat-enter fallback |
| `PLAYER_REGEN_ENABLED` | clear `plInCombat`, set dirty | Combat-exit |
| `COOLDOWN_VIEWER_DATA_LOADED` | `ScheduleRebuild()` | CDM data loaded |
| `COOLDOWN_VIEWER_SPELL_OVERRIDE_UPDATED` | `ScheduleRebuild()` | CDM spell override |
| `CooldownViewerSettings.OnDataChanged` | `ScheduleRebuild()` | CDM settings change |

### Area 5: OnUpdate Handler (new)

**Add:** A per-frame OnUpdate handler on the root frame (or a dedicated lightweight frame), matching `Hunter.Update()`:

```lua
local lastPolledTime = -1
local function OnUpdateHandler(self, elapsed)
    local now = GetTime()
    if now == lastPolledTime then return end
    lastPolledTime = now
    PollPackLeader()
end
```

Wire in `RefreshVisibility()`: when `isPackLeaderActive` becomes true, set `root:SetScript("OnUpdate", OnUpdateHandler)`; when false, set `root:SetScript("OnUpdate", nil)`.

## Risk Assessment

| Risk | Severity | Mitigation |
|------|----------|------------|
| CDM frame pool recycling causes stale refs | Medium | Lazy re-resolve + 3 CDM rebuild event triggers (same as Azor) |
| Persistence migration breaks existing saved predictions | Low | Dual-read: prefer `DB.plNextBeastId`, fall back to `DB.nextIndex`; one-time migration |
| GetTime() throttle misses rapid multi-beast spawns within single frame | Low | Same as Azor — if Azor handles it, PLBeast handles it |
| Removing C_UnitAuras fallback loses edge cases | Low | Azor has never used C_UnitAuras for beast ready buffs; CDM is the authoritative source |
| OnUpdate handler cost when isPackLeaderActive but idle | Low | GetTime() equality check makes it a single comparison per frame; same cost as Azor |
| Removing login/boss-pull reset changes user expectations | Medium | Documented in ROADMAP SC6; self-correcting model auto-fixes on first beast spawn |

## Implementation Order Recommendation

1. **Add new state variables + data tables** (plPhase, plHasCdBuff, PL_TALENT, PL_COUNTDOWN, expand TRACKED_SPELL_IDS)
2. **Add new CDM functions** (CDMFrameHasAura, ScheduleRebuild, MatchTrackedSpell, TryCache, ScanDeep, SaveState/RestoreState)
3. **Rewrite BuildCDMCache** to use new scan pattern
4. **Write PollPackLeader** — the core self-correcting prediction function
5. **Wire OnUpdate handler** in RefreshVisibility
6. **Update event registrations** (add CDM events, remove UNIT_AURA, update spec change handling)
7. **Update ADDON_LOADED** persistence migration
8. **Remove dead code** (timer poll, CheckAuraState, prevReady, snapshot-diff, C_UnitAuras fallback, rebuild-on-all-false)
9. **Remove login/boss-pull reset** (PLAYER_ENTERING_WORLD isInitialLogin block, ENCOUNTER_START block)
10. **In-game verification** — side-by-side with AzortharionUI
11. **Performance measurement** — AddonProfiler comparison
